\# Tasks API



REST API для управления списком задач, реализованное на языке Go. Проект демонстрирует применение принципов SOLID, Dependency Injection, работу с HTTP-протоколом, JSON‑сериализацией и потокобезопасным in‑memory хранилищем.



\## Особенности реализации



\- \*\*Архитектура\*\*: чёткое разделение ответственности (модели, хранилище, хендлеры).

\- \*\*Dependency Injection\*\*: хранилище внедряется через конструктор `handlers.New()`.

\- \*\*Потокобезопасность\*\*: in‑memory хранилище использует `sync.RWMutex` для безопасной работы с данными из нескольких горутин.

\- \*\*Единый формат ответов\*\*: все ответы возвращаются в формате JSON с соответствующими HTTP‑статусами.

\- \*\*Тестируемость\*\*: использование интерфейсов (`storage.Storage`) позволяет легко подменять реализацию хранилища (например, на Mock для тестов).

\## Установка и запуск



1\. Инициализируйте модуль (если ещё не сделан):

&#x20;  ```bash

&#x20;  go mod init tasks-api

Запустите сервер:

bash

cd cmd/server

go run main.go

Сервер запустится на порту 8080.

Доступные эндпоинты

Метод	Путь	Описание	Статус успеха

GET	/health	Проверка работоспособности сервиса	200 OK

GET	/tasks	Получить список всех задач	200 OK

POST	/tasks	Создать новую задачу	201 Created

GET	/tasks/{id}	Получить задачу по ID	200 OK / 404 Not Found

PUT	/tasks/{id}	Обновить задачу по ID	200 OK / 404 Not Found

DELETE	/tasks/{id}	Удалить задачу по ID	204 No Content / 404 Not Found

Важно: Для путей с {id} подставляйте числовой идентификатор задачи (например, /tasks/1).

Технические детали

Обработка времени

Поле created\_at хранится в структуре как тип time.Time, но при сериализации в JSON автоматически преобразуется в строку формата RFC 3339 (ISO 8601). Время всегда фиксируется в UTC.



Потокобезопасность

Реализация MemoryStorage использует sync.RWMutex:



RLock/RUnlock используется при чтении списка задач или одной задачи.

Lock/Unlock используется при создании, обновлении и удалении, чтобы избежать состояния гонки (race condition) при работе с мапой.

Единый формат ошибок

Все ошибки возвращаются в едином формате:



json

{"error": "текст ошибки"}

Соответствующий HTTP‑статус устанавливается автоматически через вспомогательную функцию writeJSON.



Логирование

Сервер логирует каждый входящий запрос в консоль в формате:

\[METHOD] /path

Например: \[POST] /tasks или \[GET] /health.



Также логируются ошибки сериализации JSON, если клиент разорвал соединение во время отправки ответа.



Расширяемость

Благодаря использованию интерфейса storage.Storage, вы можете легко заменить текущее in‑memory хранилище на любую другую реализацию (например, PostgreSQL), не меняя код хендлеров. Достаточно реализовать методы интерфейса для работы с базой данных и передать новый экземпляр в handlers.New().
